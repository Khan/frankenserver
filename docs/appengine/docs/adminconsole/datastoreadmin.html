Administering Your Datastore (Experimental)

You can use the Datastore Admin tab of the Admin Console to copy entities between applications, or delete them, in bulk. You can select all entities of a kind, or all entities of all kinds. This functionality is simple to enable and use in your application's Administration Console.

  • Copying Entities to Another Application
  • Deleting Entities in Bulk

Copying Entities to Another Application

You can use the Datastore Admin tab of the Admin Console to copy all entities of a kind, or all entities of all kinds, to another application. The datastore copy functionality uses the Datastore Admin screen in the Data section of the source application's Admin Console. From the Datastore Admin screen, you can select and copy entity kind(s) with the click of a button:



Prerequisites

Datastore Admin is available only if it is enabled in the source app's Admin Console. The application must be built in Python.

A Note for Java Developers

The datastore copy feature is currently available only for Python applications. If your target app is built in Java, you'll need to create a non-default Python runtime for it and use that as the target application. The following steps describe how to create a non-default Python runtime for your target application using a sample application called datastore_admin:

  1. Download Python 2.5.
  2. Download the datastore_admin app.
  3. Download the Python SDK.
  4. Grant permission for the source application to write to the target application
  5. Run appcfg.py -A <your_app_id> update <directory-of-demo-app>

Procedure

Copying a datastore involves five basic steps:

  1. Enable datastore_admin in the source application
  2. Enable remote_api in the target application
  3. Grant permission for the source application to write to the target application
  4. Set the source application to read-only mode (recommended)
  5. Copy your data

If you used the demo app described in A Note for Java Developers, you can skip steps 1–3.

1. Enable datastore_admin in the Source Application

The datastore copy feature relies upon the Datastore Admin tab in the source application's Administration Console. You can enable Datastore Admin via the Applications Settings page of the Administration Console. Under the Built-Ins heading, click Enable next to Datastore Admin:



If you want to enable the datastore_admin at a custom URI, specify that path in app.yaml using the url handler instead of the builtin handler:

- url: /datastore_admin
  script: $PYTHON_LIB/google/appengine/ext/datastore_admin/main.py

This maps the datastore_admin handler to the URL /datastore_admin. You can replace /datastore_admin with any URL you choose.

]

2. Enable remote_api in the Target Application

In order for the target application to receive data from the source, you need to enable its remote_api builtin by adding the following to app.yaml:

builtins:
- remote_api: on

3. Grant Permission to Write to the Target App

To grant permission for the source app to write to the target app, include the following in the appengine_config.py file in the root directory of your application: 

remoteapi_CUSTOM_ENVIRONMENT_AUTHENTICATION = ('HTTP_X_APPENGINE_INBOUND_APPID', ['source appid here'])

If you do not have such a file, you can create a new one or copy the sample located in google/appengine/ext/appstats/sample_appengine_config.py.

After editing the file, upload the modified version of your application using appcfg.py update.

4. Set the Source Application to Read-only Mode

Copying entities into a new datastore takes time. Writes done during the copy might not be transferred. For these reasons, we highly recommend setting the source application to read-only mode.

If the source application is not in read-only mode, you can still copy data, but you'll see a notreadonly warning in the Admin Console. The destination datastore is not guaranteed to receive a complete copy of the new data unless writes are disabled in source application.

5. Copy your data

Once you've configured your source and target applications, you can select the entity kind(s) to copy individually or in bulk, and copy them using the Copy To Other App button. On the confirmation screen, enter the remote endpoint of the target app:

  • The typical remote endpoint is http://your_target_app_id.appspot.com/_ah/remote_api.
  • If you used the sample app to copy your data, the remote endpoint is http://datastore-admin.your_target_app_id.appspot.com/_ah/remote_api.
  • If you used an alternate major version, the remote endpoint of your target application is http://app_version.your_target_app_id.appspot.com/_ah/remote_api.

The system validates the request. If the remote_api connection can be established, one or more mapreduce operations begins to copy the data. You can follow the link to see the status of the initial set of mapreduce operations. If the application uses the namespace feature, a mapreduce runs for each namespace for each entity kind.

You can view a summary of the copy status from the datastore admin page. To view the individual mapreduce status, you can visit http://your_target_app_id.appspot.com/_ah/mapreduce/

Caveats

  • This feature is currently experimental.
  • Copying entities in bulk happens within your applications, and thus counts against your quota.
  • Once you've copied entities, you may wish to re-enable writes in your source application.
  • Copying entities overwrites all entities with the same keys in the target datastore. You can perform the copying operations multiple times without the risk of creating duplicates. If you are performing multiple copy operations, be aware that the copy function does not delete extra data in the target datastore.
  • Most source applications use a Master/Slave Datastore. It is possible to copy data if the source app uses High Replication, but the copy may not include recently saved entities. If you attempt to copy from a high replication datastore, you'll see a high_replication_warning error in the Admin Console. While it is possible copy from a High Replication source app, Google does not currently support this use case.

Deleting Entities in Bulk

You can use the Datastore Admin tab of the Admin Console to delete all entities of a kind, or all entities of all kinds, in the default namespace. To enable this feature, simply enable Datastore Admin in the Administration Console.

Adding this builtin enables the Datastore Admin screen in the Data section of the Administration Console. From this screen, you can select the entity kind(s) to delete individually or in bulk, and delete them using the Delete Entities button.

Warning! Deleting entities in bulk happens within your application, and thus counts against your quota.

This feature is currently experimental. We believe this feature is currently the fastest way to bulk-delete data, but it is not yet stable and you may encounter occasional bugs.
